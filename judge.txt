01
What you are building
You are building a Claude Code plugin called an SEO Command Center. It is not a chatbot and not a notebook. It is an autonomous agent that does the work a technical-SEO analyst does by hand: read a crawl, find every problem, rank them by impact, and produce the fixes.

The pipeline your plugin must run
Stage	What it does to the Screaming Frog export
1. Ingest	Read internal_all.csv (and the issue files), normalize the columns, report how many URLs were loaded.
2. Detect	Apply the SEO rulebook to find every issue: missing / duplicate / too-long / too-short titles and meta descriptions, missing / multiple H1s, broken links (4xx), server errors (5xx), redirects (3xx), redirect chains and loops, missing image alt text, thin content, orphan pages, non-indexable-but-linked pages, canonical problems.
3. Prioritize	Give each issue a severity and an impact score, rank them, and count how many URLs each affects.
4. Fix champion	Use the local model to rewrite bad / missing titles and meta descriptions within the length limits, and build a redirect map for broken links.
5. Deliver	A live dashboard while it runs, plus report.json (machine-readable), a client-ready report.html, and the fix files (a CSV of new titles / metas, a redirect map).
Why a plugin and not just a script? A Claude Code plugin is a single installable bundle: a skill (the orchestrator), sub-agents (the workers), a local MCP server (your engine + a live dashboard on localhost), and a slash command (e.g. /seo-audit export/). The starter bundle already wires all of this. You implement the SEO logic.

Two surfaces it serves
The operator (you / an analyst with Claude Code): runs the command, watches the live cockpit at http://localhost:7700 as issues populate.
The client (anyone with a browser): receives the exported report.html file. They never touch Claude Code. The localhost dashboard is the cockpit; the exported report is the deliverable.
02
The input: a Screaming Frog export
Screaming Frog SEO Spider is the industry-standard site crawler. It crawls a website and exports the results as CSV / Excel. You do not run Screaming Frog and you do not need the internet to crawl — we crawled the sites for you (on a licensed, unlimited copy) and hand you the export files. Your agent reads those files locally.

What an export folder contains
internal_all.csv — the master file. One row per URL, with every column. This is your primary input.
Issue files — Screaming Frog's own Bulk Export → Issues → All dump: separate CSVs per issue type (e.g. page_titles_missing.csv, response_codes_client_error_4xx.csv). Useful to cross-check your detection, but your agent must be able to derive issues from internal_all.csv itself.
The real columns you work with (from internal_all.csv)
Column	Meaning
Address	The URL.
Status Code	HTTP code: 200 ok, 3xx redirect, 4xx broken, 5xx server error.
Indexability / Indexability Status	Indexable or Non-Indexable, and why (noindex, canonicalised, blocked, etc.).
Title 1, Title 1 Length, Title 1 Pixel Width	The page title and its length / pixel width.
Meta Description 1, Meta Description 1 Length	The meta description and its length.
H1-1, H1-1 Length, H2-1	First H1, its length, first H2.
Word Count	Body word count (thin-content signal).
Crawl Depth	Clicks from the start URL.
Inlinks / Outlinks	Internal links into / out of the page (0 inlinks = orphan).
Response Time	Seconds to respond (slow-page signal).
Canonical Link Element 1	The canonical URL declared by the page.
Meta Robots 1	noindex / nofollow directives.
Content Type	text/html, image, pdf, etc.
Redirect URL / Redirect Type	Where a 3xx points and the redirect type (301 / 302).
Screaming Frog v24 can detect 300+ issues across many tabs (Page Titles, Meta Description, H1/H2, Response Codes, Canonicals, Directives, Images, Content, Hreflang, and more). You are not expected to detect all 300 — the rulebook lists the issues you must detect.

05
The SEO rulebook
You need no prior SEO knowledge. Every issue below is a rule over the columns. Detect these. The grader's ground truth is built from the same rules, so precise detection scores high.

Issue	Rule (from the columns)	Severity
Missing title	Title 1 empty on an indexable 200 page	High
Duplicate title	same Title 1 on 2+ indexable URLs	High
Title too long	Title 1 Pixel Width > 561, or Title 1 Length > 60	Medium
Title too short	Title 1 Length < 30	Low
Missing meta description	Meta Description 1 empty on an indexable 200 page	Medium
Duplicate meta description	same meta on 2+ indexable URLs	Medium
Meta description too long	Meta Description 1 Length > 155	Low
Missing H1	H1-1 empty on a 200 page	Medium
Multiple / duplicate H1	more than one H1, or H1 duplicated across pages	Low
Broken link (client error)	Status Code 400–499	High
Server error	Status Code 500–599	High
Redirect	Status Code 300–399	Medium
Redirect chain / loop	a URL redirects to a URL that itself redirects (chain), or back to itself (loop)	High
Missing image alt text	image rows with empty alt attribute	Medium
Thin content	Word Count < 200 on an indexable page	Low
Orphan page	Inlinks = 0 on an indexable 200 page	Medium
Non-indexable but linked	Indexability = Non-Indexable yet has inlinks	Medium
Slow page	Response Time > 1.0 second	Low
The exact thresholds (pixel widths, character limits) are published here on purpose: they make grading objective. The full rulebook.md in the bundle adds edge-case notes (e.g. only count duplicates among indexable pages, ignore non-HTML content types for title checks).

06
The output contract
Your run must end by writing outputs/report.json in exactly this shape. The grader reads this file. If it is missing or malformed, you lose the points that depend on it.

{
  "site": "example.com",
  "urls_crawled": 1240,
  "summary": {
    "total_issues": 318,
    "by_severity": { "High": 42, "Medium": 180, "Low": 96 }
  },
  "issues": [
    {
      "type": "missing_title",
      "severity": "High",
      "affected_urls": ["https://example.com/a", "https://example.com/b"],
      "count": 12,
      "explanation": "12 indexable pages have no title tag."
    }
  ],
  "fixes": {
    "titles": [
      { "url": "https://example.com/a", "old": "", "new": "Buy Blue Widgets Online | Example" }
    ],
    "redirect_map": [
      { "from": "https://example.com/old", "to": "https://example.com/new", "reason": "404 -> closest live page" }
    ]
  },
  "recommendations": [
    "Fix the 42 High-severity issues first: 12 missing titles and 30 broken internal links."
  ],
  "run_meta": { "model": "qwen3.5:9b", "model_calls": 18, "duration_sec": 372 }
}
Field rules: issues[] is required and is what the leaderboard scores against ground truth; every issue needs type, severity, affected_urls, count. The fixes block is required for the champion tier. run_meta.model_calls is used for the efficiency score. The full JSON Schema is report.schema.json in the bundle — validate against it before you submit.

08
Your build process is recorded and judged
Read this carefully. We do not judge only the final result. We judge how you built it. Part of your score comes from the process your Claude Code session followed — how you decomposed the problem, how you used sub-agents and tools, how you debugged, and how disciplined your commits were.

The files you MUST maintain and commit (this is graded)
Your repo must contain these process, context and memory files. They are how we judge how you worked with the AI, not just what it produced. Some are recorded automatically by the starter bundle; the rest you keep as you go. Missing or faked files cost you points (section 09 gate).

File	What it is & why it matters	How
.claude/audit.jsonl
auto · logs	Process audit log. Every tool call, command, file edit, sub-agent run and result, in order. The tamper-evident spine of your process — shows your real decomposition, tool use and debugging.	The bundle's hooks (SessionStart, UserPromptSubmit, PreToolUse, PostToolUse, SubagentStop, Stop) write it automatically. Keep .claude/settings.json in place and commit the file.
agent-log.md
auto · transcript	Full session transcript. Every prompt you wrote, every decision the model made, every tool result. Shows your prompting craft and how you steered the agent.	Claude Code writes it to ~/.claude/projects/<project>/. Export at the end with the bundle's scripts/export-transcript.sh and commit it.
CLAUDE.md
context/memory	Your project memory / instructions file. The context you gave the agent: the goal, constraints, conventions, the rulebook summary, "always/never" rules, gotchas you discovered. This is the single clearest signal of good practice — strong builders engineer their context, they don't just chat.	Create CLAUDE.md at the repo root. Update it as you learn. Claude Code loads it automatically each session.
PROMPTS.md
memory	Your key prompts log. The handful of prompts that actually moved the build — the ones you iterated on, the system/sub-agent prompts, the "this finally worked" prompt. Shows how you think and direct an agent.	Keep a running markdown list. Paste the prompt, one line on what it was for, and whether you had to revise it.
DECISIONS.md
memory	Your decision / learnings log. A short running note: what you tried, what failed and why, what you changed. This is your engineering judgement on the record — it is what separates a builder from a button-presser.	Append a 1–2 line entry whenever you make a real choice or hit/fix a wall. Timestamps help.
.claude/settings.json
config	Your hooks config. Keeps the audit log recording. Proves your audit.jsonl is genuine (its events must line up with your git timestamps and transcript).	Ships in the bundle. Leave it in place; do not disable the hooks.
git history
logs	Incremental commits with meaningful messages. The human-readable spine of your process. ≥10 commits spread across the build, not one 6-hour dump.	Commit after each working step. Write real messages ("add redirect-chain detector", not "update").
The bundle ships starter templates for CLAUDE.md, PROMPTS.md and DECISIONS.md — fill them in as you build. Three independent records (the audit log, the transcript, and your git history) must agree; that is how we tell a real, legible process from a fabricated one.

What the judges look for in your process
Signal	What good looks like
Decomposition	You broke the build into clear steps and committed at each — not one 6-hour dump.
Agentic architecture in action	The audit log shows real sub-agent delegation and tool use, not one giant prompt doing everything.
Debugging discipline	When the model produced a bug or a tool failed, the log shows you caught it, diagnosed it, and fixed it — that is the core skill we test.
Verification	You ran your own tests / re-ran the audit and checked output before declaring done.
Honesty	The transcript and git history match your demo. No after-the-fact edits to fake a clean story.
Do not edit or fabricate the logs. A single giant commit, a missing audit.jsonl, or a transcript that contradicts your code caps your score (see section 09 gate). The point is not surveillance — it is that the best builders have a clean, legible process, and we reward it.
